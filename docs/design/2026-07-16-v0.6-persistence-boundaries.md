# Wandas 0.6 persistence boundaries

- Status: Accepted
- Date: 2026-07-16

## Decision

WDF and Recipe JSON are independent artifacts with different responsibilities.

### WDF 0.4

WDF stores one concrete analysis result. Version 0.4 maps one strict xarray Dataset
through the `h5netcdf` engine. It writes the raw Frame tensor once as `data`, channel
metadata as channel-sized variables, stable channel IDs and represented dimensions as
xarray coordinates, and typed constructor state as strict JSON in root attributes.
An explicit codec registry covers `ChannelFrame`, `SpectralFrame`, `SpectrogramFrame`,
`CepstralFrame`, `CepstrogramFrame`, `NOctFrame`, and `RoughnessFrame`.

The root state contains the exact Frame type and constructor parameters needed to
reconstruct its analysis domain. Semantic dimensions live only in `data.dims` and are
validated against the reconstructed Frame. Version 0.4 is the only supported WDF
schema; versions 0.1 through 0.3 and future versions fail without fallback or migration.

`SpectralFrame` and `SpectrogramFrame` always represent the complete canonical
one-sided frequency axis. Frequency and local-time values are derived from constructor
state instead of being duplicated in WDF. Frequency-axis slicing is outside this
contract; band-limited work uses shape-preserving filters, plot ranges, or explicit
band-analysis APIs.

WDF retains display `operation_history` as a source prefix. It does not restore live
semantic lineage, operation objects, or a Dask graph.

### Recipe schema 2 JSON

`RecipePlan.save()` and `RecipePlan.load()` persist reusable operation intent in a
standalone `.recipe.json` file. The artifact contains named inputs and the portable
operation graph, but never Frame samples or Dask tasks. It uses the existing strict
`wandas.recipe` schema version 2 and refuses unknown schema versions or malformed JSON.

A typical evidence pair is:

```text
analysis.wdf
analysis.recipe.json
```

The first is a concrete typed result; the second is a reusable processing plan. WDF
does not embed or reference Recipe JSON in 0.6, keeping their compatibility schedules
independent. Analyzer can consume either artifact without a Wandas-specific GUI API.

## Compatibility rules

- Readers accept only WDF 0.4; every other version fails explicitly.
- WDF root attributes, data variables, coordinates, and Recipe schema 2 require exact fields.
- Runtime-only or unregistered operations fail Recipe extraction; they are not silently
  converted to display history.
- Saving passes raw Dask data directly to xarray without first materializing the full
  tensor as NumPy. Loading returns a backend-backed Dask array; the source path must
  remain available until computation or persistence completes.

## 日本語要約

WDF は型付き解析結果、`.recipe.json` は再利用可能な処理手順です。WDF は現行の 0.4 のみを読み、全 built-in Frame の型・軸・解析パラメータを復元します。0.1〜0.3 と future version の fallback は持ちません。保存はraw Dask tensorをxarrayへ直接渡し、load後のtensorは元WDFをbackendとして遅延評価されます。SpectralFrame と SpectrogramFrame は完全な one-sided 周波数軸だけを表し、Recipe は WDF に埋め込みません。
